Part 0 — Pre-flight (on production)

0.1  Load the dry-run seed data  DATABASE

Supabase → SQL Editor (or VPS pooler). Run cc-notes and stuff/sql/seed_dryrun_data.sql. It is idempotent (fixed dd000000-… UUIDs) — safe to re-run; a teardown block sits at the bottom. Loads a partner firm, a COO matter (buyer+seller), a BC matter, a PRC/RCF matter, an enquiry thread, and a council POC.

0.2  Create the two test logins  AUTH

Logins can't be seeded from SQL (auth is Supabase-owned). Supabase → Authentication → Users → Add user, with these exact emails + a password + Auto confirm. The handle_new_user() trigger links them to the seeded profile rows and preserves the role on first sign-in.

Client logindryrun.client@example.com → lands on /dashboard, sees the BC matter (Thabo Molefe)
Partner logindryrun.partner@sterlinghayes.co.za → lands on /partner, sees the COO matter
Want to test reset/invite emails? example.com is a black hole. Swap both emails (in the seed file AND the Auth user) to inboxes you control. Branded auth email itself is still gated on Resend SMTP/DNS (see the gate) — until then Supabase sends from its default sender with a throttle.

0.4  Confirm MFA before staff sign in  SECURITY

MFA (TOTP) is mandatory for staff/admin. The first staff sign-in is forced to enrol at /auth/mfa. Sanity-check this works before the run, or staff lock out.

Lockout recovery (keep handy) If a staff account can't get past the challenge: DELETE FROM auth.mfa_factors WHERE user_id='<uid>'; — then they re-enrol on next login. Do NOT disable TOTP at the project level or every staff account locks out at once.

0.5  Sample upload files

Reuse any small file for uploads — e.g. from ~/Downloads/Change of Ownership_Examples. Limits: PDF / JPG / PNG / WebP, ≤ 50 MB. Files go to the private Supabase Storage bucket matter-documents (signed URLs, not Drive).

Part 2 — Scenario walkthroughs

A · Change of Ownership — end to end

partner → staff

The headline: two parties, one matter, per-party docs on Supabase Storage.

  1. As partner → Refer a matter. Service = Change of Ownership → form splits into Seller (shown first) + Buyer sections. Fill both (buyer is a business → contact-person fields). Add a partner file ref. Submit → matter created with status new.
  2. Click Upload supporting documents now → COO onboard form shows shared docs once (transfer letter, deed search, clearance figures) + per-party sections (ID per party; COR 14.3 for business; COC electrical for seller+COT). Upload to each or mark Not available + reason. Accept POPIA consent → Submit.
  3. As staff → /admin/matters → open the COO matter. Verify Parties card (seller before buyer, contact fields, no banking fields), per-party docs with working View / Download (signed URLs), auto-renamed files <Owner>_<Doc>, the COO stage selector + phase bar, and status control (new → open).
  4. On a party card use Create account from party → make a CRM contact OR a portal login (temp pw banner). Use Rename on a document.
Pass: COO matter has both parties (banking absent) · per-party signed View/Download work · uploads in Storage not Drive · stage/phase/status controls operate · create-account + rename work.

B · Property Rates Clearance (RCF)

partner / staff
  1. As partner → Refer a matter → Service = Property Rates Clearance. The RCF / RCC / RCA selector appears — pick RCF (RCA/RCC show a notice + block submit). Seller-details section shows above the entity section. Enter rates account no. (+ query ref for COJ). Submit.
  2. Open the onboard link → municipality-specific RCF doc set renders (COT = memo screenshot; COJ/COE = CQI screenshot; council statement required; meter readings optional). Upload + consent + submit.
  3. As staff → open the matter → Service details card shows subtype RCF + the service_data (rates acct, query ref). Confirm the seeded Khumalo RCF matter also renders.
Pass: RCF subtype + muni fields captured · RCA/RCC blocked with notice · muni-specific docs render · service-details card populated.

C · Single-client onboarding + Storage

staff → client
  1. As staff → /admin/matters/new → create a client + BC matter (matter title auto-builds MUNI_SVC_CLIENT_PROPERTY). Copy the onboarding link, or use Collect documents on the matter detail (mints/reuses a 7-day link anytime).
  2. Open the onboard link (prefilled from the client row) → fill details + upload FICA docs → Submit.
  3. Matter detail → docs show View / Download via signed Storage URLs. Test the staff Upload document button and Rename.
Pass: link mints + prefills · uploads land in Supabase Storage · link marked used · staff upload + rename work.

D · Enquiries / comms hub + Call
